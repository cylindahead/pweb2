Low-Level Systems Programming — C

I got into C through the kernel. You can't seriously study Linux internals without reading C, and once you start reading it seriously you realise it's not the hostile, arcane thing its reputation suggests. It's a small language. The complexity isn't in the language — it's in what the language lets you get close to. Memory, pointers, the actual layout of data in physical RAM. That closeness is what makes it interesting.

  • Why C specifically. Higher-level languages give you abstractions that are useful for building things fast. C gives you almost none, which means you have to build your own and understand what's underneath them. When you write a struct in C, that struct is literally just bytes at an offset. When you dereference a pointer, something is actually happening in memory. You can't wave your hands at any of it. That discipline makes you better at every other language because you stop taking the abstractions for granted.
  • What I'm working through. Kernel source reading is the main thing — mm/ and fs/ mostly. Outside of that: a memory allocator, a simple ring buffer, a basic parser that handles its own memory. Nothing impressive. The point is to write C in situations where the decisions matter and you can't delegate them to a library.
  • Pointers are not complicated. They have a reputation they don't deserve. A pointer is a number. That number is a memory address. Dereferencing it means "go to that address and read what's there." The problems people have with pointers aren't about pointers — they're about not having a clear mental model of what their program's memory looks like at any given moment. Build that model and pointers follow naturally.
  • Memory management. Manual memory management is where C programmers get things wrong most often. Not because the concept is hard — allocate, use, free — but because in a real program the distance between allocation and free is large, and anything that touches that memory in between is an opportunity for a bug. Reading how the kernel handles this — the slab allocator, RCU for read-heavy data structures, careful ownership at every step — is the best education on the problem I've found. The kernel cannot afford a garbage collector. It has developed extremely precise discipline instead.
  • Tools that matter. valgrind for memory errors. gdb for understanding what is actually happening at runtime. strace for watching how your program talks to the kernel. perf for profiling. The C tooling is old, stable, and very good. Most people don't use it enough.

Linux Kernel Development

The kernel is the most consequential piece of software most people will never think about. It runs on everything. It's all readable. You can clone the entire history right now. That combination — massive consequence, total transparency — is rare and worth taking seriously.

  • It's not written the way I expected. I expected dense, impenetrable expert-only code. What I found was C that reads almost like documentation if you give it time. The comments are often better than those in professional codebases I've worked in. The people who wrote this wanted it to be understood — because they needed others to review it, maintain it, and fix it after they moved on.
  • The OOM killer has opinions. The out-of-memory killer decides which process to kill when memory runs out and can't continue. I expected: kill the biggest process. What I found in mm/oom_kill.c scores processes on size, runtime, lineage, the oom_score_adj knob, and whether killing them would free enough memory to matter. It has a concept of "badness." It logs extensively before it acts.
  • VFS is one of the best abstractions I've seen. A network socket, a pipe, a device, a regular file — all expose the same interface because they all implement the same file_operations struct. The kernel doesn't care what's underneath. That abstraction is why you can pipe sensor output into a file the same way you'd pipe text.
  • Build setup. Debian host, QEMU guest, minimal config from make localmodconfig. Change something, compile, boot the guest, see what happens. If the guest panics I delete it. Never test kernel changes on a machine you need.
  • The goal. Get a patch accepted upstream. It will be small — a documentation fix, a checkpatch.pl cleanup, a missing error check. The size isn't the point. Learning the submission process is: git format-patch, scripts/get_maintainer.pl, sending to the right list, responding to review. That's the actual skill.

Open Source

  • What changed my mind about contributing. Reading through the commit history of a project I use constantly, I found a commit that fixed a bug I had hit myself and worked around with a hack. The author had two commits to their name total. One of them made my life measurably better. I deleted my hack that same evening.
  • How I approach a new codebase. Read the git log before the issues. Not the full history — enough to see what kinds of changes get made, who makes them, what the commit style is, how maintainers give feedback. Most first contributions get rejected not because the code is wrong but because the contributor skipped this step.
  • Why I keep configs public. Writing something as if someone else might read it produces better output than writing purely for yourself. Private repos invite sloppiness you spend twice as long untangling later.
  • What I'm watching. bcc and bpftrace for eBPF tooling. A handful of security and data engineering projects I use regularly and might eventually have something useful to add to.

Hardware & PC Building

  • Current setup. Dell Inspiron 14 5410 2-in-1 running Debian stable. Not exciting hardware but completely reliable, which matters more. KVM/QEMU for a Kali VM and a Windows VM. External SSD handles the VM images. The real bottleneck is 16GB of RAM shared across everything. I am always running too many VMs.
  • The PS3 situation. I still have a PS3. The Cell processor was a strange and ambitious design — seven Synergistic Processing Elements alongside a PowerPC core, built for parallel floating-point workloads. Two directions I'm considering: jailbreak it with custom firmware and use it as a retro gaming box, or strip the board and build a PC inside the PS3 chassis. The case has deliberate airflow and the dimensions could fit a small ITX board. The jailbreak is the easier path. The chassis build is the more interesting one. I'll probably end up doing both on separate machines.
  • The home lab I'm planning. A tower that runs 24/7 without noise or stupid power draw, with enough RAM to stop rationing VMs. Second-gen Ryzen Threadripper workstation platforms on the used market for the PCIe lane count. ECC RAM mandatory — non-ECC on a machine that runs continuously is a bet you will eventually lose.
  • What building teaches you. Specs lie. The bottleneck is almost always somewhere you didn't look first. Cooling is almost always the real constraint. This is also true of software. Hardware just makes it visceral because the wrong answer produces heat you can feel.

Cyberdecks

A cyberdeck is a handmade portable computer. The form factor is whatever the builder decides. The enclosure is custom. The OS is Linux. The point is that it is yours in a way no off-the-shelf laptop can be. I found this community through Ube Boobey on YouTube — her builds are what made me take this seriously as something I could actually do rather than just admire. The way she documents the process, including the parts that don't work and the redesigns, is more useful than any polished build guide.

  • The build I want to make. Portable Linux terminal. Raspberry Pi 5, 7-inch 1080p display angled for lap use, custom 40% mechanical keyboard PCB with hot-swap sockets and USB-C output, PETG printed frame with acrylic panels, 10,000mAh battery for four hours minimum. Weight target under 900 grams. Folds shut so the keyboard protects the display. Software: minimal Debian, no desktop environment, tmux, neovim. A machine for one thing. Constraints are features.
  • The keyboard problem. Getting a proper mechanical keyboard into a cyberdeck form factor cleanly means designing a custom PCB. That means learning KiCad properly, which I haven't done yet. KiCad is the long lead-time item on this entire project. Everything else I can figure out as I go.
  • Why not just use a laptop. A laptop is optimised for the average user doing average tasks. The cyberdeck is a machine with no compromises in the directions I care about and zero concern for the ones I don't. That trade-off is the point, not a limitation.
  • Community. r/cyberDeck has hundreds of documented builds. The ones worth studying are not the most visually striking — they're the ones where the design decisions are clearly functional. Aesthetic without function is cosplay.

Gaming

  • Holstin. A Polish survival horror game with fixed camera angles and a visual style that leans into body horror deliberately rather than gratuitously. The atmosphere is genuinely oppressive. What I find interesting about it is that the controls and camera are part of the design — they make you feel as vulnerable as the protagonist is supposed to be. That's not a flaw, it's the point. Not many games are willing to use the interface itself as part of the horror.
  • Call of Duty: Modern Warfare 2. The opposite end of the spectrum from Holstin and I don't feel the need to justify playing both. The multiplayer is fast, the gunplay is tight, and there's something satisfying about a game that is extremely good at one narrow thing and doesn't pretend to be anything else. The map design in MW2 is studied seriously by people who design competitive multiplayer games. Favela, Terminal, Highrise — these maps have been analysed the way architectural layouts get analysed. Three-lane structure, sightline control, choke points. There is more design intelligence in a COD map than most people give it credit for.
  • The PS3 and what I want to do with it. See the hardware section. Short version: jailbreak for the full library, or build a PC inside the chassis. Probably both, on separate machines.
  • Gaming on Linux. Proton has made this genuinely viable. Most things run. Some run better than on Windows. When something breaks it's annoying and always educational — you end up understanding what DXVK is, how Vulkan translation layers work, how the Windows API maps to Linux syscalls. More than you wanted to know. Worth knowing anyway.
  • Emulation. RPCS3 emulating the Cell processor required reverse-engineering an architecture Sony and IBM designed to be hard to compete with. The team did it anyway, in spare time, for free. That kind of stubborn technical project is genuinely inspiring independent of whether you use the result.

Vinyl & Deep House

My main reference point right now is Jazzman's Deep Inspiration Show on NTS Radio. It's a show where you don't know what you're going to hear and that's the whole point. The selections move between deep house, jazz-influenced electronic music, soul, and things that don't fit neatly into any category. What connects them is a mood — patient, textured, music that takes up space without demanding attention. I've found more music through that show than through any algorithm. That's not a coincidence. A human with taste surfaces things that recommendation systems don't reach.

  • Why vinyl. A record requires a decision. You pick something, take it out, put it on, commit to a side. You can't shuffle. When it ends you have to get up. That friction is doing work. Removing friction from music consumption produced a world where people have access to everything and listen carefully to almost nothing. I'd rather own fifty records I know deeply than have access to fifty million I've scrolled past.
  • Learning to DJ with Strudel. Strudel is a live coding environment for music built in JavaScript that runs in the browser. You write patterns in a mini-language and they play in real time — change the code, the music changes immediately. It's the most direct connection I've found between programming and music. The feedback loop is tight enough that it feels more like programming than music practice, which is probably why it's working for me. I'm using it to understand how rhythm and structure work in electronic music from the inside — not by listening and analysing, but by building patterns and hearing immediately what works and what doesn't. If you write code and want to understand music better, this is the entry point I'd recommend.
  • What deep house actually sounds like. Slower than you think — 118 to 124 BPM. Chord progressions from jazz and gospel that don't resolve where pop music expects them to. Space in the mix. A bassline that is melodic rather than purely rhythmic. The feeling it produces at the right volume on a good sound system does not translate to phone speakers. It requires some investment of attention to give back what it has.
  • Where to start. Jazzman's Deep Inspiration Show on NTS is the answer to this question — a better introduction than any playlist because it comes with a selector who knows why each record is there. For the originals: Larry Heard as Mr. Fingers is the beginning of the thread. Follow that to early Trax Records and DJ International out of Chicago. Discogs is the only reliable database for tracking down older pressings.
